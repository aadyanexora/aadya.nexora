Aadya – Nexora AI
Current Application Status
===========================

Architecture:
- Backend: FastAPI application structured under `backend/app` with modular routers
  (auth, chat, admin).
- Database: PostgreSQL container; SQLAlchemy ORM with models for users, documents,
  document_chunks, conversations, messages.
- Embeddings: Local SentenceTransformers (all-MiniLM-L6-v2) used via
  `OpenAIService.get_embeddings`.
- Vector store: FAISS persisted under a docker volume; vector metadata maps
  (doc_id, chunk_index).
- Chat: Groq API streaming for responses; multi-turn context and RAG retrieval
  implemented.
- Frontend: Next.js 14 (App Router, TypeScript) with Tailwind CSS for styling.

Key Features:
- JWT authentication with admin check; tokens include a `role` claim.
- **Access/refresh token flow:** login/register issues a short‑lived access
  token (15 min) plus a 7‑day refresh token. Refresh tokens are hashed and
  stored in `refresh_tokens`; they can be rotated or revoked via
  `/auth/refresh` and `/auth/logout`.
- **Credit-based usage tracking:** every user starts with 1000 credits; tokens
  consumed by chat requests are deducted from this balance. Monthly credit
  top‑ups and remaining balances are viewable by administrators.
- Document ingestion with automatic chunking (500-token chunks, 50-token
  overlap), metadata tracking (source, filename, page) and PDF/text support.
- Admin-only ingestion endpoints with multipart/form-data payloads, robust
  error logging, and optional metadata; JSON body ingestion temporarily
  reverted during debugging.
- Alembic migrations `0003_add_chunk_metadata.py` and
  `0004_add_usage_tracking.py` added for new schema changes.
- Conversation history persisted; endpoints to list conversations and fetch
  history.
- Streaming chat endpoint requiring JWT and returning conversation id,
  `context_meta` array, structured citations (`sources` field) and footnotes.
  Input + output tokens are counted and logged; cost estimates recorded.
- Search and retrieval now log embedding and query timings; vector hits return
  metadata and are persisted in FAISS.
- Structured logging with request IDs and Prometheus metrics (/metrics).
- Rate limiting via slowapi (10 requests/min on chat streaming; keyed per
  user when authenticated).
- Health endpoint (/health) and Docker-compose service healthchecks.
- Multi-stage Dockerfiles for lean production images; Torch 2.0 CPU installed.
- Alembic migration framework initialized with an initial migration script.  
  * env.py now adjusts `sys.path` for module imports and overrides
    `sqlalchemy.url` using the `DATABASE_URL` env var to avoid
    interpolation errors when running inside Docker.


Development Notes:
- Logging config moved to `app/core/logging.py`.
- Rate limiter configured in `main.py`; chat route decorated with limiter.
- Frontend styled using Tailwind; chat and dashboard pages upgraded.
- Docker-compose includes volumes for Postgres and FAISS persistence.

    newer Docker Compose releases.

- Additional tooling: python-multipart, slowapi, alembic added to requirements.

Repo housekeeping / Termux move
-----------------------------
- A cleanup script was added at `scripts/clean_caches.sh` to remove `__pycache__`,
  `*.pyc`, frontend `.next` and `node_modules` before packaging the repository
  for transfer to Termux or other constrained environments.
- `.gitignore` updated to ignore common cache and build artifacts.

Instructions for running migrations:
  cd backend
  alembic upgrade head    # includes new migration adding chunk metadata

Notes about Termux:
- Building heavy native dependencies (Torch, FAISS) is rarely feasible directly
  in Termux. Recommended approach is to either use prebuilt wheels, perform
  heavy builds on a remote machine and copy artifacts, or use remote embedding
  services while running the lightweight backend in Termux.
